Ad Hoc vs Playbook

Same modules — different packaging. Know when to probe vs when to commit automation.

1. Decision Tree — Click a path
Will you run this exact change more than once?
NO — one-time / exploratory
↓
Ad hoc command
YES — repeatable / production
↓
Playbook in Git
Need handlers, roles, or vault?
↓
Always playbook
Just checking connectivity or facts?
↓
Ad hoc is fine
Start here: Ad hoc is for probes and quick fixes. Playbooks are for anything you want reviewed, versioned, and repeated safely.
2. Side-by-Side Comparison

Ad hoc command

  • Single module, single invocation
  • Great for ping, uptime, fact gathering
  • Quick package install to test sudo
  • Not stored in Git by default
  • No handlers or role support
  • Syntax: ansible <pattern> -m <module> -a "args"

Playbook

  • Multiple tasks, ordered execution
  • Handlers, variables, templates, roles
  • Idempotent, reviewable in pull requests
  • CI/CD pipeline integration
  • Vault for secrets
  • Syntax: ansible-playbook site.yml
3. Equivalent Examples (same module, different packaging)
# Ad hoc
ansible webservers -i inventory/hosts.ini.local -m ansible.builtin.ping

# Playbook equivalent (playbooks/ping.yml)
---
- name: Connectivity check
  hosts: webservers
  tasks:
    - name: Ping all web servers
      ansible.builtin.ping:
# Ad hoc — install tree on one host (-b for become/sudo)
ansible web1 -i inventory/hosts.ini.local -b \
  -m ansible.builtin.apt -a "name=tree state=present update_cache=yes"

# Playbook equivalent
---
- name: Install tree
  hosts: web1
  become: true
  tasks:
    - name: Ensure tree package
      ansible.builtin.apt:
        name: tree
        state: present
        update_cache: true
# Ad hoc — check apache status
ansible webservers -i inventory/hosts.ini.local -b \
  -m ansible.builtin.service -a "name=apache2 state=started"

# Playbook (labs/playbooks/apache.yml) — full install + handler
ansible-playbook -i inventory/hosts.ini.local playbooks/apache.yml
# Ad hoc — run uptime
ansible webservers -i inventory/hosts.ini.local \
  -m ansible.builtin.command -a "uptime"

# Note: prefer ansible.builtin.command over raw shell
# Use ansible.builtin.shell only when pipes or redirects needed
# Ad hoc — gather distribution facts
ansible web1 -i inventory/hosts.ini.local \
  -m ansible.builtin.setup -a "filter=ansible_distribution*"

# Facts auto-gathered in playbooks (gather_facts: true by default)
# Disable with gather_facts: false for speed on large fleets

6. Migration Path: Ad Hoc → Playbook

When an ad hoc command works, promote it to a playbook:

  1. Run ad hoc successfully on one host
  2. Create YAML play with same module and arguments
  3. Add name: to each task for readable output
  4. Add handlers if service restart needed
  5. Extract to role when reused across projects
  6. Commit to Git and run from CI

11. Parallel Execution
# Default forks=5 — run against 5 hosts in parallel
ansible webservers -i inventory/hosts.ini.local -m ansible.builtin.ping

# Serial (one host at a time) — useful during rolling fixes
ansible webservers -i inventory/hosts.ini.local -m ansible.builtin.ping -f 1

Playbooks use the same -f flag. Canary deploys combine --limit web1 then full fleet.

12. Ansible Configuration Precedence

ansible.cfg in labs/ sets defaults when you run commands from that directory:

roles_path = roles
