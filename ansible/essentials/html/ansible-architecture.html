Control Node

The control node is where you install Ansible and run commands. It holds your playbooks, inventory, roles, and vault files. It does not need to be a server — your laptop or a CI runner works. Ansible pushes configuration outward; managed nodes never pull from a central Ansible server.

📋 Example: First connectivity check

Goal: Verify Ansible can reach all web servers before running playbooks.

Command: ansible webservers -i inventory/hosts.ini -m ansible.builtin.ping

Flow: Control node reads inventory → resolves web1/web2 IPs → opens SSH → runs ping module → returns pong per host.

Result: Both hosts report SUCCESS with "ping": "pong" ✅

2. Core Components (click to view details)
Required

Inventory

Lists hosts, groups, and connection variables (ansible_host, ansible_user).

Automation

Playbook

YAML file describing desired state: plays, tasks, handlers, roles.

Execution

Modules

Units of work with idempotent behavior. Use FQCN: ansible.builtin.apt.

Config

ansible.cfg

Project defaults: inventory path, roles_path, host_key_checking.

Packaging

Collections

Bundle modules, plugins, roles. ansible.builtin is the core collection.

Security

Ansible Vault

Encrypt secrets at rest in YAML files; decrypt at runtime with password.

# labs/inventory/hosts.ini
[webservers]
web1 ansible_host=10.0.1.10 ansible_user=ubuntu
web2 ansible_host=10.0.1.11 ansible_user=ubuntu

[webservers:vars]
ansible_python_interpreter=/usr/bin/python3

[dbservers]
db1 ansible_host=10.0.1.20 ansible_user=ubuntu
# labs/playbooks/apache.yml
---
- name: Install and configure Apache
  hosts: webservers
  become: true
  tasks:
    - name: Install apache2
      ansible.builtin.apt:
        name: apache2
        state: present
        update_cache: true

    - name: Enable mod_rewrite
      ansible.builtin.apache2_module:
        name: rewrite
        state: present
      notify: Restart apache2

  handlers:
    - name: Restart apache2
      ansible.builtin.service:
        name: apache2
        state: restarted
# Ad hoc module examples (FQCN recommended)
ansible webservers -m ansible.builtin.ping
ansible web1 -b -m ansible.builtin.apt -a "name=curl state=present"
ansible webservers -b -m ansible.builtin.service -a "name=apache2 state=started"
ansible web1 -m ansible.builtin.template -a "src=motd.j2 dest=/etc/motd"

# labs/ansible.cfg
[defaults]
inventory = inventory/hosts.ini
roles_path = roles
host_key_checking = False
interpreter_python = auto_silent
retry_files_enabled = False

[privilege_escalation]
become_method = sudo
# Fully Qualified Collection Name (FQCN)
ansible.builtin.ping          # core collection
ansible.builtin.apt           # package management (Debian/Ubuntu)
ansible.builtin.service       # systemd/sysv service control
community.general.docker_container  # from community collection

# List installed collections
ansible-galaxy collection list
# Encrypt secrets for Git storage
ansible-vault encrypt vault/secrets.yml --vault-password-file ~/.vault_pass

# Playbook loads encrypted vars
vars_files:
  - ../vault/secrets.yml

# Run with decryption
ansible-playbook playbooks/nodejs.yml --vault-password-file ~/.vault_pass
3. Typical Workflow (from zero to running playbook)
1

Provision infrastructure

Terraform (or cloud console) creates VMs with private IPs, security groups allowing SSH from control node.

2

Configure inventory

Copy hosts.ini.local.example to hosts.ini.local and set real private IPs and ansible_user.

3

Test connectivity

ansible webservers -i inventory/hosts.ini.local -m ansible.builtin.ping — all hosts must return pong.

4

Run playbooks

ansible-playbook -i inventory/hosts.ini.local playbooks/apache.yml — apply desired state idempotently.

5

Version control and CI/CD

Commit playbooks to Git; run ansible-playbook from pipeline with vault password from secrets manager.

4. Agentless vs Agent-Based CM
Aspect Ansible (agentless) Agent-based tools
Software on targets SSH + Python only Daemon/agent installed and maintained
Firewall Port 22 (SSH) inbound from control Custom agent port + central server
Bootstrap Works immediately if SSH works Install agent before first config
Scale model Parallel forks from control node Agents pull or server pushes
Windows WinRM instead of SSH Often native agent
5. Parallel Execution (forks)
Control Node
forks=5 default
⇉
web1
task 1..n
web2
task 1..n
db1
task 1..n

Ansible runs tasks in parallel across hosts (default 5 forks). Each host progresses through the play independently. Tune with -f 10 or forks = 10 in ansible.cfg. Serial execution uses serial: in the play.
