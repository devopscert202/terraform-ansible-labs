Playbook Execution & Handlers

Tasks run in order; handlers run once at the end of the play — only if notified

Repository file: labs/playbooks/apache.yml — installs apache2, enables mod_rewrite, notifies handler to restart Apache.

2. Execution Timeline (apache.yml)
1
Play starts — hosts: webservers, become: true
Ansible connects to web1 and web2 in parallel (forks)
2
Task: Install apache2 (ansible.builtin.apt)
First run: changed=1. Second run: changed=0 (idempotent)
3
Task: Enable mod_rewrite + notify: Restart apache2
If module reports changed, handler is queued — NOT executed immediately
4
End of play — all tasks complete on all hosts
Handler block runs now, in order listed in handlers:
H
Handler: Restart apache2 (ansible.builtin.service)
Runs only if notified AND task reported changed. Runs once per host even if notified multiple times.
—
Second playbook run (nothing changed)
mod_rewrite already enabled → task ok, not changed → handler NOT notified → no restart
3. Playbook Structure
---YAML document start (required at top)
name:Human-readable play description
hosts:Inventory pattern (webservers, all, etc.)
become:true escalates via sudo
tasks:List of modules to execute in order
handlers:Tasks that run on notify at play end
4. Code Examples
# labs/playbooks/apache.yml
---
- name: Install and configure Apache
  hosts: webservers
  become: true
  tasks:
    - name: Install apache2
      ansible.builtin.apt:
        name: apache2
        state: present
        update_cache: true

    - name: Enable mod_rewrite
      ansible.builtin.apache2_module:
        name: rewrite
        state: present
      notify: Restart apache2

  handlers:
    - name: Restart apache2
      ansible.builtin.service:
        name: apache2
        state: restarted
# Handler rules (memorize these)
# 1. Handlers run at END of play (not immediately)
# 2. Handlers run only if notifying task reports CHANGED
# 3. Same handler notified N times → runs ONCE per host
# 4. Handler order in handlers: block is preserved
# 5. Handlers can notify other handlers (avoid loops)

# Run playbook
ansible-playbook -i inventory/hosts.ini.local playbooks/apache.yml
# notify must match handler name exactly
notify: Restart apache2

handlers:
  - name: Restart apache2    # ← this string must match

# Alternative: listen (multiple notifiers)
handlers:
  - name: restart web service
    listen: "restart apache"
    ansible.builtin.service:
      name: apache2
      state: restarted

# Task can use: notify: "restart apache"
# Force handlers to run mid-play (rare)
- name: Apply config
  ansible.builtin.template:
    src: apache.conf.j2
    dest: /etc/apache2/apache2.conf
  notify: Restart apache2

- name: Flush handlers before next task
  ansible.builtin.meta: flush_handlers

- name: Verify site responds
  ansible.builtin.uri:
    url: http://localhost/
    status_code: 200
5. Idempotency — First vs Second Run

First run (fresh Ubuntu)

web1  : ok=3  changed=2  failed=0
web2  : ok=3  changed=2  failed=0

# apt installed apache2 (changed)
# mod_rewrite enabled (changed) → handler runs

Second run (no drift)

web1  : ok=2  changed=0  failed=0
web2  : ok=2  changed=0  failed=0

# apt: already present (ok)
# mod_rewrite: already on (ok)
# handler: NOT notified
6. YAML Pitfalls
MistakeFix
Tabs in YAMLUse spaces only (2 spaces typical)
Handler name typonotify string must match handler name exactly
Missing becomeAdd become: true for apt/service tasks
Wrong hosts patternVerify with ansible-inventory --graph
7. ansible-playbook Flags
FlagPurpose
--syntax-checkValidate YAML without running
--checkDry run (module-dependent)
--diffShow file changes
--limit web1Subset of hosts
-v to -vvvvVerbose debugging
8. Extended Handler Patterns

listen: Multiple notifies can trigger one handler via shared listen name.

flush_handlers: Meta task runs handlers immediately mid-play (rare).

Role handlers: roles/webserver/handlers/main.yml — same notify rules as play handlers.
