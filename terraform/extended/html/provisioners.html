Provisioners & Lifecycle Hooks

Provisioners run scripts during resource lifecycle events. They are a last resort — prefer cloud-init, images, or configuration management tools.

Overview State Provisioners Functions Projects
Repository lab paths:
  • terraform/extended/labs/lab04-local-exec-provisioner/ — Run commands on the machine executing Terraform
  • terraform/extended/labs/lab05-remote-exec-provisioner/ — SSH into a reachable host and run commands
When NOT to use provisioners: Software installation, ongoing config drift, application deployment, secrets distribution, or anything requiring idempotent CM at scale. Use Ansible, cloud-init, Packer, or native cloud services instead.

local-exec runs where you run terraform apply — your laptop or CI agent.

Example: Bootstrap notification after resource creation

Good use: Write a timestamp file locally after creating a marker resource (Lab 04).

Bad use: Install nginx with apt-get via remote-exec on every apply — use an AMI or Ansible role.

remote-exec caveat: Lab 05 connects to an existing host you supply. Terraform does not create that host.

3. Code examples (tabs)
resource "terraform_data" "local_action" {
  input = var.message
  provisioner "local-exec" {
    command = "printf '%s\n' '${self.input}'"
  }
}
resource "terraform_data" "bootstrap" {
  input = var.host
  connection {
    type        = "ssh"
    host        = var.host
    user        = var.user
    private_key = file(pathexpand(var.private_key_path))
  }
  provisioner "remote-exec" {
    inline = ["echo Terraform remote-exec connected to $(hostname)"]
  }
}
resource "aws_instance" "example" {
  # ...
  lifecycle {
    create_before_destroy = true
    prevent_destroy       = false
    ignore_changes        = [tags["LastReboot"]]
  }
  provisioner "local-exec" {
    when    = destroy
    command = "echo cleanup hook — use with extreme care"
  }
}

4. Failure behavior

Provisioner fails on create

Resource is tainted. Next apply retries creation. Partial infrastructure may exist.

on_failure = continue

Suppresses failure (rare). Resource marked created even if script failed. Use only with strong observability.

Destroy provisioner

Runs while dependencies are being torn down. Network routes and security groups may already be gone.

Connection retries

SSH may not be ready immediately. Use timeouts and health checks, or wait for cloud-init completion out-of-band.

8. Glossary
  • Provisioner — Script hook on lifecycle events
  • connection — Transport config for remote provisioners
  • Taint — Mark resource for recreation on next apply
  • self — Reference to the resource containing the provisioner
9. SSH connectivity checklist (Lab 05)
  1. Security group allows TCP 22 from your IP (not 0.0.0.0/0 in production)
  2. Instance has public IP, or you reach it via VPN/bastion
  3. OS user matches AMI (ec2-user, ubuntu, admin)
  4. Private key file mode chmod 600
  5. Manual ssh -i key user@host hostname succeeds before apply

11. FAQ

Run on plan?

No — provisioners execute only during apply on create/destroy/replace.

Idempotent?

You must write idempotent scripts; Terraform does not retry logic for you.

Ansible?

Preferred for multi-step configuration; keep Terraform for infrastructure graph.
