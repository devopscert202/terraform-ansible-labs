Remote State

S3 backends, locking, workspaces, migration, and remote state consumers.

Team Terraform requires remote state in shared storage with locking. Extended labs 06–11 walk through S3, workspaces, and consumption patterns.

Core Model

Why Remote State

Collaboration

Multiple engineers and CI jobs share one state object.

S3 backend

Store state in a versioned S3 bucket with encryption at rest.

Locking

S3 native lockfile or DynamoDB prevents concurrent apply corruption.

Workspaces

Multiple state files in one backend — env separation without duplicate code.

Migration

terraform init -migrate-state moves local state to remote.

Remote state data

terraform_remote_state reads outputs from another root module.

Backend

S3 Backend Topology

Repo lab files: terraform/extended/labs/lab07-s3-backend/
backend.hcl
bucket       = "my-tf-state-unique"
key          = "extended/vpc/terraform.tfstate"
region       = "us-east-1"
use_lockfile = true
encrypt      = true
Partial config in main.tf
terraform {
  backend "s3" {}
}
Init with backend file
terraform init -backend-config=backend.hcl
