Remote State & Shared Ownership

Terraform state maps configuration to real infrastructure. Store it remotely, lock concurrent writes, name keys deliberately, and consume outputs through narrow interfaces.

Example: Two engineers, one state file

Scenario: Alice runs terraform apply while Bob runs terraform plan on the same backend key.

With locking: Bob's plan waits or fails fast with a lock error — state is not corrupted.

Without locking: Last write wins; resources may be orphaned or re-created unpredictably.

3. Backend Configuration (Code tabs)
terraform {
  required_version = ">= 1.5.0"
  backend "s3" {}
  # Backend settings cannot use input variables.
  # Pass them via -backend-config or a reviewed backend.hcl file.
}

resource "terraform_data" "state_owner" {
  input = "shared-state"
}
bucket       = "your-unique-terraform-state-bucket"
key          = "extended/dev/network/terraform.tfstate"
region       = "us-east-1"
encrypt      = true
use_lockfile = true
# Syntax check without touching remote state:
terraform init -backend=false
terraform validate

# Connect to S3 backend:
cp backend.hcl.example backend.hcl   # edit bucket/key first
terraform init -backend-config=backend.hcl

# Migrate existing local state:
terraform init -backend-config=backend.hcl -migrate-state
data "terraform_remote_state" "network" {
  backend = "local"  # or "s3" in production
  config = {
    path = var.network_state_path
  }
}

output "upstream_outputs" {
  value = data.terraform_remote_state.network.outputs
}
4. Migration Workflow
Backup local state
→
Review destination key
→
init -migrate-state
→
state list / plan

Never delete locks or edit terraform.tfstate by hand to resolve conflicts. Identify the active operation first, then coordinate with the lock holder.

Security controls
  • Enable S3 versioning for rollback
  • Encrypt at rest (encrypt = true)
  • Least-privilege IAM for state bucket
  • Block public access on the bucket
  • State may contain secrets — treat as sensitive
  • Never commit backend.hcl with secrets
Anti-patterns
  • Sharing one state key across unrelated stacks
  • Using remote state as a general inventory API
  • Disabling locking in team environments
  • Storing state in application artifact buckets
  • Manual JSON edits to fix drift
5. DynamoDB vs S3 Native Lockfile

S3 use_lockfile (Terraform 1.11+)

Lock stored alongside state in S3. Simpler setup — no DynamoDB table. Used in these labs via use_lockfile = true.

DynamoDB table (legacy pattern)

Separate lock table with dynamodb_table attribute. Still common in existing platforms. Choose one locking strategy per bucket.

11. Backend prerequisites script
# Platform team typically pre-creates:
# - S3 bucket with versioning + encryption
# - Block public access
# - IAM policy for operators

aws s3api create-bucket --bucket YOUR-UNIQUE-STATE-BUCKET --region us-east-1
aws s3api put-bucket-versioning --bucket YOUR-UNIQUE-STATE-BUCKET \
  --versioning-configuration Status=Enabled
aws s3api put-public-access-block --bucket YOUR-UNIQUE-STATE-BUCKET \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

Training labs assume this bucket exists before terraform init -backend-config.

12. State sensitivity examples

May contain

Private IPs, resource IDs, LB DNS names, database endpoints, TLS cert ARNs.

Often redacted in CLI

Passwords marked sensitive in provider schema — still may exist in state file.

Never do

Attach state bucket to public website hosting or log shipping without encryption.

13. Practice scenarios

Scenario A: New team member clones the repo

They run terraform apply without init -backend-config and use local state. Fix: follow Lab 07 two-phase init and migrate.

Scenario B: CI and laptop conflict

Pipeline holds lock; developer sees lock error. Fix: wait for CI, or coordinate force-unlock after verifying job death.
