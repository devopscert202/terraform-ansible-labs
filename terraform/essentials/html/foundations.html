Terraform Essentials

Terraform Foundations

Providers, resources, HCL syntax, and the init workflow that every lab depends on.

Terraform is a declarative infrastructure-as-code tool. You describe desired state in HCL files; Terraform compares that to current state and proposes a plan of API calls to converge reality.

Core Model

What Is Terraform?

Declarative IaC

You write resource "aws_instance" "web" blocks describing what should exist. Terraform figures out create, update, or destroy operations.

Providers

Plugins such as hashicorp/aws translate HCL into cloud API calls. Version constraints live in required_providers.

State

Terraform stores a mapping between configuration addresses and real resource IDs in a state file. Without state, plans are unreliable.

Root module

Every lab directory is a root module — a folder with main.tf (and friends) that Terraform runs as one unit.

Lock file

.terraform.lock.hcl pins exact provider versions after terraform init. Commit it in team projects.

Credential chain

Never put access keys in provider blocks. Use AWS_PROFILE, environment variables, or IAM roles.

Terraform Essentials Lab 01 uses only the Random provider for apply — AWS provider is declared for credential-chain practice.
Execution Model

How Terraform Talks to Cloud APIs

Repo lab files: terraform/essentials/labs/lab01-providers-init/main.tf
Lab directory layout
terraform/essentials/labs/lab01-providers-init/
├── main.tf              # required_providers, provider, resources
├── .terraform/          # created by init (gitignored)
└── .terraform.lock.hcl  # provider version pins
required_providers block
terraform {
  required_version = ">= 1.5.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.0"
    }
  }
}
Provider block (no secrets)
provider "aws" {
  region = var.aws_region
  # Credentials from AWS_PROFILE / env / IAM role — never here
}
First Run

Lab 01 Initialization Workflow

1

Verify toolchain

terraform version — confirm Terraform 1.5+ before working with lab files.

2

Change to lab directory

cd terraform/essentials/labs/lab01-providers-init — each lab is an isolated root module.

3

Inspect configuration

Read main.tf — note required_providers and the random_pet resource.

4

Initialize

terraform init downloads providers and writes .terraform.lock.hcl.

5

List providers

terraform providers shows resolved provider sources and versions.

6

Validate

terraform validate checks syntax and internal references — no cloud calls.

7

Optional apply

terraform apply creates the random pet; terraform destroy cleans up.

See lab01-providers-init.md for step-by-step validation criteria.
Install

Installing Terraform on Your Workstation

macOS — Homebrew
brew tap hashicorp/tap
brew install hashicorp/tap/terraform
terraform version
Linux — HashiCorp apt repo
wget -O- https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform
terraform version
Verify in lab directory
cd terraform/essentials/labs/lab01-providers-init
terraform init
terraform validate
AWS credential check (optional)
export AWS_PROFILE=your-lab-profile
aws sts get-caller-identity

Decision Guide

Terraform vs Other IaC Approaches

AspectTerraformCloudFormationAnsible
ParadigmDeclarative graphDeclarative AWS-onlyProcedural tasks
StateExplicit state fileAWS-managed stacksNone (by default)
Multi-cloudYes — provider ecosystemAWS onlyYes — agentless SSH/API
Drift detectionterraform planStack drift detectionAd hoc / check mode
Learning curveMedium — HCL + stateMedium — YAML/JSONMedium — YAML playbooks
Best fitInfrastructure provisioningAWS-native teamsConfiguration management
