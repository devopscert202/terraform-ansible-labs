destroy: Removes the resource; state returns to empty

3. Reading Plan Output
+ create
Resource does not exist yet; Terraform will create it.
~ update in-place
Attribute changes that can be applied without replacement.
- destroy
Resource removed from config or marked for deletion.
No changes
Real infrastructure matches configuration.
4. Command Reference (Code Tabs)
# Run after clone or when providers change
cd terraform/essentials/labs/lab03-plan-apply-destroy
terraform init

# Expected tail output:
# Terraform has been successfully initialized!
terraform plan

# Plan: 1 to add, 0 to change, 0 to destroy.
#
# Changes to Outputs:
#   + generated_value = (known after apply)
#
# Do you want to perform these actions?
# (plan is read-only — it does not change infrastructure)
terraform apply

# Review the plan, then type: yes
#
# Apply complete! Resources: 1 added, 0 changed, 0 destroyed.
#
# Outputs:
# generated_value = "abcdefghijkl"
terraform destroy

# Plan: 0 to add, 0 to change, 1 to destroy.
# Type yes to confirm
#
# Destroy complete! Resources: 0 added, 0 changed, 1 destroyed.
# Rewrite .tf files to canonical style
terraform fmt -recursive .

# No output = already formatted
# File names printed = files were reformatted
terraform validate

# Success! The configuration is valid.

# Run AFTER init; catches type errors and
# invalid references before plan

7. Workflow Concepts

Execution Plan

Terraform saves plan files with -out=tfplan for reviewed applies.

Targeted Operations

-target=resource limits scope — use sparingly, not in production.

Auto-approve

-auto-approve skips prompt — OK in CI with gated plan artifacts.

Refresh-only

terraform apply -refresh-only updates state without changing resources.

Common Mistakes

❌ Running apply without reading the plan first

❌ Skipping destroy on paid AWS resources after labs

❌ Running validate before init (providers not installed)

✅ Always: init → validate → plan → apply → verify outputs → destroy
